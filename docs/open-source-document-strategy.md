# 오픈소스 문서 전략

Semiconductor Education은 기능이 많은 교육 플랫폼보다, 기준이 선명한 하나의 오픈소스 문서를 목표로 합니다. 방향은 Toss Frontend Fundamentals처럼 특정 분야의 핵심 원칙을 꾸준히 정리하고, 누구나 읽고 기여할 수 있는 공개 지식 기반을 만드는 것입니다.

## 핵심 정의

이 프로젝트는 “반도체를 배우는 웹사이트”이기 전에 “반도체를 이해하는 방식을 정리한 오픈소스 문서”입니다.

- 글이 제품의 중심입니다.
- UI는 글을 더 잘 읽히게 만드는 도구입니다.
- 기능은 문서 이해를 돕는 경우에만 추가합니다.
- 기여는 코드보다 콘텐츠 품질, 출처 검증, 설명 개선을 우선합니다.

## 닮고 싶은 점

Toss Frontend Fundamentals에서 참고할 방향은 특정 UI를 복제하는 것이 아니라 문서 운영 방식입니다.

| 기준 | 적용 방향 |
| --- | --- |
| 원칙 중심 | 반도체 개념을 단순 요약이 아니라 판단 기준으로 설명 |
| 짧은 문단 | 초보자도 포기하지 않게 한 문단에 하나의 생각만 담기 |
| 구체 예시 | DRAM, HBM, EUV 같은 실제 산업 키워드로 설명 |
| 비교 구조 | 빠름/용량/전력/수율/패키징처럼 trade-off를 드러내기 |
| 공개 문서 | 누구나 읽고, 고치고, 출처를 제안할 수 있는 구조 유지 |

## 문서의 제품 원칙

### 1. 하나의 기준 문서처럼 읽혀야 한다

글이 흩어진 블로그처럼 보이면 안 됩니다. 사용자가 처음 들어와도 다음 흐름을 따라갈 수 있어야 합니다.

1. 내 레벨을 확인한다.
2. 로드맵으로 전체 지도를 본다.
3. 핵심 글 3편을 읽는다.
4. 용어 사전으로 막힌 단어를 푼다.
5. 복습 질문으로 자기 말 설명을 확인한다.

### 2. 쉬움과 깊이를 분리하지 않는다

기초 글과 심화 글을 완전히 나누면 초보자는 깊이에 도달하지 못하고, 심화 학습자는 배경 설명을 놓칩니다. 따라서 각 글은 쉬운 결론에서 시작해 실무 판단으로 끝나야 합니다.

### 3. 공식 자료를 읽는 힘을 길러야 한다

공식 자료를 요약하는 것이 목표가 아닙니다. 목표는 학습자가 Samsung Semiconductor, SK hynix Newsroom 같은 공식 자료를 읽고 핵심 주장, 기술 의미, 제약 조건을 분리할 수 있게 만드는 것입니다.

### 4. 기능은 문서를 방해하지 않아야 한다

로그인, 댓글, 좋아요, 조회수 같은 기능은 MVP에서 제외합니다. 문서 신뢰도와 가독성을 높이지 않는 기능은 우선순위를 낮춥니다.

## 오픈소스 기여 단위

기여는 다음 단위로 받는 것을 기준으로 합니다.

| 기여 유형 | 예시 |
| --- | --- |
| 설명 개선 | 어려운 문장을 더 쉽게 바꾸기 |
| 출처 보강 | 공식 자료 링크 추가, 확인일 갱신 |
| 용어 추가 | 용어 사전에 쉬운 설명과 실무 맥락 추가 |
| 복습 질문 추가 | 정의 암기가 아닌 설명형 질문 추가 |
| 오류 수정 | 잘못된 기술 설명, 오탈자, 깨진 링크 수정 |
| UI 개선 | 본문 가독성, 모바일 레이아웃, 목차 사용성 개선 |

## 함께 개발하는 방식

이 프로젝트는 개인 블로그처럼 완성 글만 올리는 방식이 아니라, 여러 사람이 같은 기준으로 문서를 계속 개선하는 방식을 목표로 합니다.

- 아이디어는 이슈로 제안합니다.
- 새 글은 바로 작성하지 않고 주제, 대상 독자, 공식 출처 후보를 먼저 확인합니다.
- 글은 `proposal`, `draft`, `source-check`, `review`, `ready` 상태를 거쳐 공개합니다.
- 작은 문장 수정, 출처 보강, 용어 추가는 독립 PR로 빠르게 반영합니다.
- 리뷰는 개인 취향보다 콘텐츠 기준, 출처 정책, 학습 흐름을 우선합니다.

구체적인 운영 방식은 [커뮤니티 운영 기준](community-governance.md)에 둡니다.

## 문서 품질 게이트

새 글이나 큰 수정은 다음 질문을 통과해야 합니다.

- 초보자가 첫 문단에서 주제를 이해할 수 있는가?
- 중급자가 구조와 trade-off를 확인할 수 있는가?
- 심화 학습자가 수율, 전력, 열, 패키징, 양산성 같은 제약을 볼 수 있는가?
- 공식 자료의 문장을 복사하지 않고 학습자 관점으로 재구성했는가?
- 글을 읽은 뒤 사용자가 자기 말로 설명할 수 있는 질문이 있는가?

## 저장소 운영 기준

- `main`은 안정 문서와 안정 UI만 둡니다.
- `dev`는 다음 공개 후보를 모으는 통합 브랜치입니다.
- 글, 문서, UI, CI 변경은 작업 브랜치와 PR로 분리합니다.
- 커밋은 Conventional Commits를 따릅니다.
- 큰 문서 보완은 가능한 한 “문서 인덱스”, “기획”, “출처”, “콘텐츠”처럼 작업별로 나눠 커밋합니다.

## 장기 목표

장기적으로 이 프로젝트는 반도체 입문자가 공식 산업 자료를 읽기 전 거치는 기준 문서가 되어야 합니다.

Phase 1에서는 DRAM, HBM, EUV 3편의 깊이를 높입니다. Phase 2에서는 NAND, SSD, 패키징, 공정 흐름으로 확장합니다. Phase 3에서는 직무별 읽기 경로와 면접형 설명 질문까지 연결합니다.
