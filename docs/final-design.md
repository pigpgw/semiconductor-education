# 최종 설계 확정

이 문서는 1차 MVP를 일단락하기 위한 최종 설계 기준입니다. 앞으로 기능이나 글을 늘릴 때도 이 기준을 벗어나지 않는지 먼저 확인합니다.

현재 구현 상태는 [MVP 정리](mvp-summary.md), 오픈소스 문서 운영 방향은 [오픈소스 문서 전략](open-source-document-strategy.md), 이후 보완 우선순위는 [보완 기획](planning-improvements.md)에 따릅니다. 최종 설계를 다음 구현 단위로 나누는 기준은 [설계 보완](design-supplement.md)에 둡니다.

## 제품 정의

Semiconductor Education은 로그인 없이 바로 읽는 한국어 반도체 오픈소스 문서입니다. 쉬운 설명으로 시작하지만, 최종 목표는 DRAM, HBM, EUV 같은 산업 키워드를 구조, 제약 조건, 실무 판단 기준까지 설명할 수 있게 만드는 것입니다.

이 프로젝트는 반도체 뉴스를 모으는 사이트가 아닙니다. 공식 자료를 그대로 요약하는 블로그도 아닙니다. 역할은 공식 기술 자료를 학습자가 읽을 수 있도록 해설하고, 초보자에서 실무 관점까지 올라가는 길을 설계하는 것입니다.

제품의 중심은 기능이 아니라 글입니다. UI, 레벨 진단, 용어 사전, 복습 질문은 하나의 공개 문서를 더 잘 읽히게 만드는 보조 장치입니다.

## 핵심 사용자

| 사용자 | 필요한 것 | 설계 대응 |
| --- | --- | --- |
| 처음 배우는 학생과 비전공자 | 용어 장벽 없는 시작점 | 한 줄 결론, 쉬운 비유, 용어 사전 |
| 취업 준비생과 전공 학습자 | 기술 키워드의 구조적 이해 | DRAM/HBM/EUV 핵심 글, 비교, 체크 질문 |
| 지방/비수도권 학습자 | 혼자 따라갈 수 있는 학습 경로 | 레벨 진단, 로드맵, 로그인 없는 학습 노트 |
| 포트폴리오 검토자 | 사고 구조와 구현 완성도 | 출처 정책, 콘텐츠 기준, 반응형 문서 UX |

## 최종 MVP 범위

1차 MVP는 많은 글보다 “설계 품질과 핵심 글의 깊이”를 증명하는 데 집중합니다.

| 화면 | 역할 |
| --- | --- |
| `/` | 프로젝트 목적, 학습 트랙, 핵심 글 진입 |
| `/level` | 자기 수준 진단과 레벨 선택 |
| `/roadmap` | 기초부터 실무 관점까지 전체 학습 순서 |
| `/learn` | 난이도와 주제별 글 목록 |
| `/learn/[slug]` | MDX 기반 본문, 출처, 다음 글 |
| `/glossary` | 검색, 카테고리, 난이도로 탐색하는 용어 사전 |
| `/practice` | 자기 말로 설명하는 복습 질문 |
| `/study` | 로그인 없는 로컬 체크리스트와 노트 |

첫 공개 글은 DRAM, HBM, EUV 3편으로 고정합니다. NAND, 패키징, 직무별 글은 다음 단계로 넘깁니다.

## 난이도 설계

난이도는 콘텐츠 접근을 막는 권한이 아니라 읽기 깊이를 추천하는 장치입니다.

| 레벨 | 읽기 목표 | 완료 기준 |
| --- | --- | --- |
| 기초 | 기술이 왜 필요한지 이해 | DRAM/HBM/EUV가 푸는 문제를 말한다 |
| 중급 | 구조와 트레이드오프 연결 | 셀, TSV, 패터닝, 대역폭을 비교한다 |
| 심화 | 실무 제약까지 판단 | 수율, 열, 전력, 고객 검증, 양산성을 함께 설명한다 |

사용자는 `/level`에서 진단을 받을 수 있고, 언제든 직접 레벨을 바꿀 수 있어야 합니다. 추천 레벨은 안내일 뿐 잠금 장치가 아닙니다.

## 글 작성 구조

모든 핵심 글은 같은 흐름을 따릅니다.

1. 한 줄 결론
2. 쉬운 비유
3. 실제 구조
4. 비교와 트레이드오프
5. 현업 키워드
6. 공식 자료 기반 근거
7. 체크 질문

쉬운 설명과 깊은 설명을 따로 분리하지 않습니다. 같은 글 안에서 독자가 자연스럽게 더 깊은 층위로 내려가게 만듭니다.

## 디자인 확정

디자인은 기술 문서 사이트답게 조용하고 밀도 있게 유지합니다. 사용자가 준 다크 문서형 레퍼런스를 기준으로 다음 요소를 반영했습니다.

- 칩 형태 로고와 sticky header
- 다크 배경, 기술 그리드, 절제된 블루/틸 포인트 컬러
- 홈 hero, stats strip, 5개 학습 트랙 카드
- 문서 프리뷰 사이드바, 핵심 콜아웃, 코드형 읽기 흐름
- 로드맵 타임라인과 최신 글 리스트
- 모바일 360px부터 읽히는 반응형 레이아웃

장식 이미지는 학습 이해를 돕지 않으면 사용하지 않습니다. 카드와 박스는 반복 항목, 도구, 문서 프리뷰처럼 실제 정보 구조를 담을 때만 사용합니다.

## 로그인 없는 정책

MVP에는 로그인, 회원가입, 소셜 로그인, 댓글, 좋아요, 조회수, 서버 기반 진도 저장을 넣지 않습니다.

학습 콘텐츠, 레벨 진단, 용어 사전, 복습 질문은 모두 공개 접근입니다. `/study`의 체크리스트와 노트는 현재 브라우저의 `localStorage`에만 저장하며, 이름, 이메일, 학교, 지역 같은 개인정보를 요구하지 않습니다.

## 출처와 신뢰 기준

핵심 주장은 공식 자료를 우선합니다.

- Samsung Semiconductor
- SK hynix Newsroom
- 필요 시 표준 문서, 논문, 기업 공식 기술 자료

공식 자료 문장을 복사하지 않습니다. “공식 표현 → 기술 의미 → 학습자 질문” 구조로 바꿔 설명합니다. 최신성이 중요한 글은 `updatedAt`과 출처 확인일을 관리합니다.

## 일부러 제외한 것

1차 MVP에서는 다음을 하지 않습니다.

- 로그인 기반 개인화
- 커뮤니티, 댓글, 좋아요
- 과도한 랜딩 페이지형 마케팅 UI
- 검증되지 않은 비공식 자료 기반 주장
- 많은 글을 얕게 늘리는 방식
- 서버 분석을 위한 개인정보 수집

## 1차 일단락 기준

현재 단계는 다음 조건을 만족하면 일단락으로 봅니다.

- 핵심 페이지 8개가 구현되어 있다.
- DRAM, HBM, EUV 핵심 글 3편이 있다.
- 레벨 진단, 용어 사전, 복습 질문, 로컬 학습 노트가 있다.
- README와 docs만 읽어도 제품 의도, 학습 설계, 출처 정책, Git workflow가 이해된다.
- `npm run lint`, `npm run typecheck`, `npm run build`, `npm run check:viewport`가 통과한다.
- 작업은 `dev` 통합 후 PR로 `main`에 반영된다.

## 다음 단계

다음 작업은 새 기능보다 콘텐츠 품질을 우선합니다.

1. DRAM/HBM/EUV 3편을 더 깊게 다듬고 출처 확인일을 점검합니다.
2. 용어 사전과 복습 질문을 늘려 혼자 공부하는 사용자의 막힘을 줄입니다.
3. NAND, SSD, 패키징, 공정 흐름은 Phase 2 이후에 추가합니다.
4. 포트폴리오용으로 “왜 이렇게 설계했는지”를 README와 문서에서 더 잘 드러냅니다.

구체적인 우선순위는 [보완 기획](planning-improvements.md)을 기준으로 합니다.
