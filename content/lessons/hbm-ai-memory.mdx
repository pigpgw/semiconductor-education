<OneLine>
HBM은 AI 연산기가 굶지 않도록 데이터를 넓은 길로 한꺼번에 보내는 고대역폭 DRAM입니다.
</OneLine>

## 쉬운 비유

AI 모델 학습은 아주 큰 주방에서 수많은 요리를 동시에 만드는 일과 비슷합니다.

GPU는 요리사입니다. 계산을 매우 빠르게 합니다. 그런데 재료가 좁은 문 하나로 조금씩 들어오면 요리사는 기다릴 수밖에 없습니다.

HBM은 재료 창고를 주방 바로 옆에 두고, 좁은 문 대신 넓은 출입구를 여러 개 만든 구조에 가깝습니다. 덕분에 GPU가 필요한 데이터를 더 많이, 더 빠르게 받을 수 있습니다.

여기서 중요한 점은 요리사의 속도만 올린다고 전체 식당이 빨라지지 않는다는 것입니다. 재료가 늦게 오면 요리사는 기다립니다. AI 시스템에서도 GPU 계산 성능이 높아질수록 메모리에서 데이터를 가져오는 길이 더 중요해집니다.

## 실제 구조

HBM은 High Bandwidth Memory의 약자입니다.

핵심은 두 가지입니다.

- DRAM 칩을 수직으로 쌓습니다.
- 쌓은 칩 사이를 TSV로 연결합니다.

평면 위에 길게 배치하는 대신 위로 쌓으면, GPU 가까이에 더 많은 메모리를 둘 수 있습니다. 또 TSV를 통해 위아래 칩을 직접 연결하면 데이터가 이동하는 길을 짧고 넓게 만들 수 있습니다.

<Term title="TSV">
Through-Silicon Via의 줄임말입니다. 실리콘 칩을 수직으로 관통하는 전극을 만들어, 쌓인 칩들이 위아래로 직접 데이터를 주고받게 하는 기술입니다.
</Term>

HBM 패키지는 보통 GPU나 AI 가속기 가까이에 배치됩니다. 여러 메모리 다이를 쌓고, 아래쪽에는 신호를 모아 전달하는 base die가 놓입니다. 이 구조는 메모리 칩을 단순히 많이 붙이는 것이 아니라, 연산기와 메모리 사이의 데이터 이동 거리를 줄이는 설계입니다.

구조를 단순화하면 다음과 같습니다.

| 층 | 역할 |
| --- | --- |
| DRAM die | 데이터를 저장하는 메모리 층 |
| TSV | 위아래 die를 연결하는 수직 통로 |
| Base die | 스택의 신호와 제어를 모으는 기반 층 |
| Interposer/package | GPU와 HBM을 짧고 넓게 연결하는 영역 |

HBM을 이해할 때 “쌓았다”에서 멈추면 부족합니다. 왜 쌓았는지, 쌓은 뒤 어떻게 연결하는지, 그 연결이 전력과 열과 수율에 어떤 부담을 주는지를 같이 봐야 합니다.

## bandwidth가 중요한 이유

메모리에서 중요한 값은 용량만이 아닙니다. AI에서는 특히 **대역폭(bandwidth)**이 중요합니다.

대역폭은 일정 시간 동안 얼마나 많은 데이터를 옮길 수 있는지를 뜻합니다.

AI 모델은 계산량도 크지만, 계산에 필요한 파라미터와 중간 데이터를 계속 읽고 씁니다. GPU의 계산 능력이 아무리 높아도 메모리에서 데이터가 늦게 오면 전체 처리량이 떨어집니다.

이 병목을 흔히 **memory wall**이라고 부릅니다.

대역폭과 지연 시간(latency)은 다릅니다. 대역폭은 한 번에 지나갈 수 있는 데이터의 양이고, 지연 시간은 요청한 데이터가 도착하기까지 걸리는 시간입니다. AI 학습과 추론에서는 거대한 데이터를 계속 움직이기 때문에 대역폭이 전체 처리량에 큰 영향을 줍니다.

## HBM이 푸는 문제

HBM은 memory wall을 완전히 없애는 마법이 아닙니다. 하지만 GPU와 메모리 사이의 데이터 길을 크게 넓혀 병목을 줄입니다.

| 관점 | 일반 DRAM 모듈 | HBM |
| --- | --- | --- |
| 배치 | 보드 위에 떨어져 배치 | GPU 가까이에 패키징 |
| 연결 | 상대적으로 긴 경로 | 짧고 넓은 연결 |
| 구조 | 주로 평면 배치 | DRAM 다이 적층 |
| 강점 | 범용성과 확장성 | 높은 대역폭과 효율 |

이 차이는 시스템 설계에서도 의미가 큽니다. 일반 DRAM 모듈은 범용 서버와 PC에서 확장성이 좋습니다. HBM은 용량을 무한히 늘리기보다, 연산기 가까이에 높은 대역폭을 집중시키는 방식입니다. 그래서 AI 가속기에서는 HBM 용량, 대역폭, 패키지 면적, 전력 예산을 함께 계산합니다.

삼성반도체의 HBM 설명은 TSV 기반 적층 기술과 초고대역폭을 강조합니다. SK하이닉스 뉴스룸은 AI 확산으로 데이터 이동량이 늘고, 메모리 계층을 시스템 수준으로 최적화해야 한다는 관점을 제시합니다.

## HBM 스펙을 읽는 순서

HBM 발표를 읽을 때는 세대명만 보면 안 됩니다. HBM2E, HBM3, HBM3E처럼 이름이 바뀌는 것보다 중요한 것은 그 세대가 어떤 병목을 얼마나 줄이는지입니다.

| 읽을 항목 | 먼저 묻는 질문 | 실무 의미 |
| --- | --- | --- |
| 적층 수 | 몇 개의 DRAM die를 쌓았는가 | 용량은 늘지만 두께, 열, 수율 부담도 커질 수 있음 |
| 대역폭 | GPU가 한 번에 받을 수 있는 데이터 통로가 얼마나 넓은가 | AI 학습과 추론 처리량에 직접 영향 |
| 용량 | 모델과 중간 데이터를 얼마나 가까운 메모리에 둘 수 있는가 | 더 큰 모델, 더 큰 배치, 캐시 전략과 연결 |
| 전력 효율 | 같은 데이터를 옮기는 데 에너지를 얼마나 쓰는가 | 데이터센터 전력과 냉각 비용에 영향 |
| 패키지 조건 | GPU, interposer, substrate와 어떻게 연결되는가 | CoWoS 같은 고급 패키징 capacity와 연결 |
| 고객 검증 | 실제 가속기와 서버에서 검증됐는가 | 양산 발표와 실제 공급 가능성을 구분하는 기준 |

이 순서로 보면 “12단 HBM3E” 같은 표현도 단순한 숫자가 아닙니다. 더 많은 die를 쌓으면 용량을 키울 수 있지만, 전체 패키지 높이와 열 방출, die별 결함 관리가 어려워집니다. 따라서 적층 수가 늘었다는 발표는 성능 홍보가 아니라 제조와 신뢰성 난이도를 함께 읽어야 합니다.

HBM은 스펙 하나로 평가하기 어렵습니다. AI 서버에서는 GPU 성능, HBM 대역폭, 패키징 capacity, 전력 예산, 납기가 함께 맞아야 실제 시스템 성능으로 이어집니다.

## 패키징이 중요해지는 이유

HBM은 단순히 DRAM 칩을 많이 쌓는 문제가 아닙니다.

쌓으면 다음 문제가 생깁니다.

- 칩 사이 연결을 정확하게 맞춰야 합니다.
- 열을 잘 빼야 합니다.
- 휘어짐과 압력을 제어해야 합니다.
- 대량 생산 수율을 확보해야 합니다.

그래서 HBM에서는 패키징 기술이 제품 경쟁력의 핵심으로 올라옵니다. 메모리 셀을 잘 만드는 것뿐 아니라, 여러 칩을 안정적으로 쌓고 연결하고 식히는 능력이 중요해집니다.

실무에서 HBM을 어렵게 만드는 요인은 보통 한 가지가 아닙니다.

| 제약 | 왜 중요한가 |
| --- | --- |
| 열 | 여러 die가 쌓이면 내부 열을 빼기 어려워짐 |
| 수율 | die 하나의 결함이 스택 전체 가치에 영향을 줄 수 있음 |
| 정렬 | TSV와 범프 연결이 정확해야 신호가 안정적임 |
| 고객 검증 | GPU, 패키지, 보드, 소프트웨어 스택과 함께 검증해야 함 |
| 공급 안정성 | AI 서버 수요가 커질수록 양산 능력과 납기가 중요해짐 |

이 때문에 HBM은 메모리 기술이면서 동시에 패키징 기술이고, 시스템 기술입니다.

## HBM이 모든 문제를 해결하지는 않는다

HBM은 매우 강력하지만 만능 메모리는 아닙니다. 대역폭을 크게 키우는 대신 비용, 패키지 면적, 공급 capacity, 열 관리 같은 제약이 따라옵니다.

AI 시스템의 데이터는 한곳에만 머물지 않습니다. 일부는 HBM에 있어야 하고, 일부는 DDR이나 CXL 메모리, SSD 같은 더 큰 계층에 있어야 합니다. 자주 쓰는 데이터는 빠른 메모리 가까이에 두고, 덜 자주 쓰는 데이터는 더 큰 저장 계층에 둡니다.

이 관점에서 HBM은 “가장 좋은 메모리”라기보다 “연산기 바로 옆에서 가장 비싼 병목을 줄이는 메모리”입니다. 그래서 실제 시스템 설계에서는 HBM 용량을 무조건 늘리는 것보다 어떤 데이터가 HBM에 있어야 하는지, 어떤 데이터는 다른 계층으로 내려도 되는지 판단합니다.

| 메모리 계층 | 강점 | 한계 |
| --- | --- | --- |
| HBM | GPU 가까이에서 높은 대역폭 제공 | 비용, 패키징, 용량 확장 제약 |
| DDR/LPDDR | 범용성과 용량 확장성 | HBM보다 대역폭과 위치에서 불리 |
| CXL 메모리 | 서버 메모리 확장과 풀링 가능성 | 지연 시간과 생태계 성숙도 고려 필요 |
| SSD/NAND | 큰 용량과 낮은 저장 비용 | 연산기 바로 옆 메모리처럼 쓰기에는 느림 |

이 표를 보면 HBM 공부가 왜 시스템 공부로 이어지는지 알 수 있습니다. AI 병목은 연산기 하나가 아니라 데이터가 계층 사이를 이동하는 방식에서 생기기 때문입니다.

## 현업 관점에서 보는 핵심

AI 인프라에서는 한 칩의 성능만으로 승부하기 어렵습니다.

GPU, CPU, 메모리, 저장장치가 함께 움직입니다. 최근에는 HBM과 SSD 사이의 새로운 메모리 계층을 표준화하려는 움직임도 있습니다. 이는 AI 추론에서 용량, 전력 효율, 데이터 이동 구조를 시스템 수준으로 최적화하려는 흐름입니다.

따라서 HBM을 공부할 때는 “빠른 메모리”보다 한 단계 더 들어가야 합니다.

- 어떤 병목을 줄이는가
- GPU 가까이에 배치하는 이유는 무엇인가
- TSV와 패키징이 왜 필수인가
- 대역폭, 용량, 전력, 열 사이의 균형은 어떻게 잡는가

SK하이닉스와 Sandisk가 언급한 HBF 같은 다음 계층의 논의도 이 맥락에서 이해할 수 있습니다. HBM은 매우 빠르지만 용량과 비용, 전력 효율에서 모든 문제를 혼자 해결하지는 않습니다. SSD는 용량은 크지만 HBM처럼 빠른 메모리 계층은 아닙니다. 그 사이를 어떻게 메울 것인지는 AI 추론 시스템에서 중요한 설계 문제가 됩니다.

따라서 HBM 공부의 끝은 제품 이름 암기가 아닙니다. AI 시스템에서 데이터가 어디에 머물고, 언제 움직이고, 그 이동 비용을 어떻게 줄이는지 설명할 수 있어야 합니다.

## 공식 자료 기반 근거

삼성반도체는 HBM을 TSV 기반 적층 기술을 적용한 초고대역폭 메모리로 소개합니다. SK하이닉스는 AI 시대에 데이터 이동량 증가와 memory wall을 중요한 문제로 설명하고, HBM과 차세대 메모리 계층이 AI 인프라 최적화에 연결된다고 설명합니다.

이 글에서는 그 내용을 학습자 관점에서 “GPU가 기다리지 않게 하는 데이터 공급 구조”로 다시 정리했습니다. HBM의 공식 설명은 성능 홍보 문구로만 읽지 않고, TSV, 적층, 패키징, 시스템 병목을 함께 보는 근거로 사용했습니다.
